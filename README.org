#+title: Courier
#+author: Lucius Chen

Courier is a lightweight HTTP client for Emacs. It sends requests defined in
=.http= files using =curl= and displays responses in structured read-only
buffers.

* Features

- Edit HTTP requests as plain text in =.http= files
- Send requests with =C-c C-c= without blocking Emacs
- View responses in a single buffer with a header-line current-view indicator and an Emacs-native timeline inspector
- Variable interpolation with ={{name}}= syntax
- Built-in test assertions for status, headers, body, timing, and size
- Preview resolved requests before sending
- Select per-buffer environments from collection =env/*.env= files with =C-c C-e=
- Pretty-print JSON responses, with =v= to toggle raw view
- Start unsaved request drafts and place them into a collection on first save
- Browse the current collection in an overview buffer with =C-c C-b=
- Jump between requests and environments with =C-c C-o=
- Open a context-aware action menu with =C-c ?=

* Requirements

- Emacs 29.1 or later
- =curl= on PATH

* Installation

Clone this repository and add it to your =load-path=:

#+begin_src elisp
(add-to-list 'load-path "/path/to/courier")
(require 'courier)
#+end_src

* Quick Start

Create a file =example.http=:

#+begin_src http
+++
name = "Get User"

[body]
type = "json"

[auth]
type = "bearer"
token = "{{token}}"

[vars]
user_id = "42"
token = "abc123"

tests = ["status == 200"]
+++

GET https://httpbin.org/get/{{user_id}}?expand=profile
Accept: application/json
#+end_src

Open it in Emacs. Press =C-c C-c= to send.

* Collections

A Courier collection is a directory marked by =courier.json=. When this file is
present, Courier treats that directory as the collection root for request
discovery and does not scan environment files above it.

Minimal =courier.json= example:

#+begin_src json
{
  "name": "My API",
  "envDir": "env",
  "defaultEnv": "local"
}
#+end_src

* File Format

A =.http= file contains one request using:

- optional TOML front matter between =+++= markers
- one raw HTTP request block after the front matter

Example:

#+begin_src http
+++
name = "Create User"
timeout = 10
follow_redirects = true

[body]
type = "json"

[auth]
type = "bearer"
token = "{{token}}"

[vars]
token = "abc123"
user_id = "42"

tests = ["status == 201"]

[scripts]
pre_request = """
(setq courier-script-request
      (plist-put courier-script-request :url "https://example.com/users/42"))
"""
+++

POST https://example.com/users?page=1
Accept: application/json
Content-Type: application/json

{
  "name": "Lucy"
}
#+end_src

Metadata belongs in TOML front matter:

| Key / Table           | Example                                | Description             |
|-----------------------+----------------------------------------+-------------------------|
| =name=                | =name = "Get User"=                    | Human-readable name     |
| =timeout=             | =timeout = 10=                         | Timeout in seconds      |
| =follow_redirects=    | =follow_redirects = true=              | Follow redirects        |
| =[body]=              | =type = "json"=                        | Body serialization type |
| =[auth]=              | =type = "bearer"=                      | Auth metadata           |
| =[vars]=              | =token = "abc123"=                     | Request-scoped vars     |
| =tests=               | =tests = ["status == 200"]=            | Test assertions         |
| =[scripts]=           | =pre_request = """..."""=              | Pre/post scripts        |

Method, URL, headers, and body stay in the raw HTTP block.

** Variables

Use ={{name}}= anywhere in the URL, headers, or body. Variables are defined
with the front matter =[vars]= table. Recursive expansion is supported up to 10
levels. Query parameters stay attached to the URL itself and use ={{var}}=
placeholders when needed.
If you prefer editing URL query params as key/value lines, use
=M-x courier-request-edit-params= or the request action menu. Courier opens a
small editor buffer and normalizes the result back into the request URL when
you apply it.

** Body Types

=[body].type= controls how Courier interprets and sends the request body.

Supported types:

- =json=
- =xml=
- =text=
- =form-urlencoded=
- =none=

Example:

#+begin_src http
+++
[body]
type = "form-urlencoded"
+++

POST https://example.com/session
Accept: application/json

username={{user}}&password={{password}}
#+end_src

Send-time behavior:

- =json= adds =Content-Type: application/json= when no explicit content type is present
- =xml= adds =Content-Type: application/xml= when missing
- =text= adds =Content-Type: text/plain; charset=utf-8= when missing
- =form-urlencoded= URL-encodes the body and adds
  =Content-Type: application/x-www-form-urlencoded= when missing
- =none= requires an empty body and sends no request body

Use =M-x courier-request-set-body-type= or =C-c ?= in a request buffer to set
the body type. In the request section view, the active body type is shown in
the header line as =Body(JSON)=, =Body(XML)=, and so on.

** Auth

=[auth]= currently supports three forms:

- bearer
- basic
- header

Courier resolves auth values through the same variable expansion path as the
rest of the request. If an explicit header with the same name already exists in
the request, that explicit header wins.

Use =M-x courier-request-set-auth-type= or =C-c ?= in a request buffer to set
the auth type. Courier supports =none=, =bearer=, =basic=, and =header= in the
editor. In the request section view, the active auth type is shown in the
header line as =Auth(None)=, =Auth(Bearer)=, =Auth(Basic)=, or
=Auth(Header)=.

** Scripts

Courier supports optional Elisp script blocks in front matter:

#+begin_src http
+++
[scripts]
pre_request = """
(setq courier-script-request
      (plist-put courier-script-request :url "https://example.com/from-script"))
courier-script-request
"""
post_response = """
(setq courier-script-response
      (plist-put courier-script-response :reason "Changed"))
courier-script-response
"""
+++

GET https://example.com
#+end_src

- =pre-request= runs after parsing and env loading but before validation and
  variable resolution.
- =post-response= runs after the response is parsed but before Courier runs
  =tests= assertions.

Scripts run as Elisp. The main bound variables are:

- =courier-script-request=
- =courier-script-response=
- =courier-script-env-vars=
- =courier-script-phase=

** Environment Files

Inside a collection, Courier reads environment files from the directory
configured by =courier.json= via =envDir=. The default is =env/= at the
collection root. Environment names come from filenames such as =local.env=,
=staging.env=, and =prod.env=. You may also use =.env= for an environment
named =default=.

If =defaultEnv= is set in =courier.json= and that environment exists, Courier
activates it automatically for the current request buffer. Otherwise, Courier
prompts with =completing-read= when multiple environments are available. The
choice remains buffer-local. Values from front matter =[vars]= override values
from environment files.

Example =env/local.env=:

#+begin_example
base_url=https://api.example.com
token=abc123
#+end_example

** Test Assertions

| Expression                          | Description                    |
|-------------------------------------+--------------------------------|
| =status == 200=                     | Status code equals             |
| =status != 404=                     | Status code not equals         |
| =header content-type contains json= | Header value contains string   |
| =body contains hello=               | Response body contains string  |
| =time < 500=                        | Response time under N ms       |
| =size < 102400=                     | Response size under N bytes    |

* Key Bindings

** Request Mode (=.http= files)

| Key       | Command                        | Description            |
|-----------+--------------------------------+------------------------|
| =C-c C-c= | =courier-request-send=         | Send request           |
| =C-c C-b= | =courier-overview=             | Open collection view   |
| =C-c C-e= | =courier-request-switch-env=   | Switch environment     |
| =C-c C-j= | =courier-request-jump-section= | Jump to request section |
| =C-c [=   | =courier-request-prev-section= | Previous main section  |
| =C-c ]=   | =courier-request-next-section= | Next main section      |
| =C-c C-m= | =courier-request-set-method=   | Change HTTP method     |
| =C-c C-n= | =courier-new-request=          | Create request draft   |
| =C-c C-p= | =courier-request-preview=      | Preview resolved form  |
| =C-c ?=   | =courier-dispatch=             | Show request actions   |
| =C-c C-l= | =courier-request-validate=     | Validate syntax        |
| =C-c C-o= | =courier-open=                 | Open request or env    |
| =C-c C-r= | =courier-rename-request=       | Rename current request |
| =C-x C-s= | =courier-request-save-buffer=  | Save draft or file     |

Use =M-x courier-new-request= or =C-c C-n= to open an unsaved draft buffer. The
draft stores =name = "Untitled N"= in front matter and uses the same value in
its buffer name, but the request editor itself starts at the request line using
=courier-default-request-method=, which defaults to =GET=. On the first
=C-x C-s=, Courier asks which collection should own the request, starting from
=courier-default-collection-directory=, which defaults to a =courier/=
directory under the current user's home directory. If the chosen directory is
not already a Courier collection, Courier offers to create one there before
saving the request into that collection's =requests/= directory.
The request-line method token is highlighted with a lightweight overlay so
=GET=/=POST=/etc. remain visually obvious while editing.
The request buffer uses a section view model with a navigation header line:

#+begin_example
Params  Body  Headers  >>
#+end_example

Use =C-c [= and =C-c ]= to switch the main request sections rendered below the
divider, or =C-c C-j= to jump to any section, including =Auth=, =Vars=,
=Tests=, =Pre-request=, and =Post-response=. The request line stays fixed as
the request identity block; the area below it is the active section view, not a
jump target into raw source text, and the current section is indicated in the
=header-line= rather than repeated as a second in-buffer title.

=Params= can still be edited directly as part of the URL, but the request-side
params views normalize those values into Courier params when you edit them.
Saving then writes them back into the request URL instead of keeping two
separate truths.

Use =M-x courier-create-collection= to create a collection explicitly. Use
=C-c C-o= inside a collection to open another =.http= file or switch the
current buffer to a different environment from the same minibuffer picker.
Use =C-c ?= in a request buffer for request actions, including the export
submenu for copying =curl=, =httpie=, or =wget= commands with or without
variable interpolation. Request content itself stays source-backed: Courier
uses transient for actions, not as the primary editor for params, auth, or
script content.

Use =C-c C-b= to open the current collection overview. The overview buffer
lists requests, shows recent status and environment context, and supports:

- =RET= to open the request at point
- =C-c ?= to show overview actions
- =n= to create a new request
- =f= to create a folder
- =r= to rename the request at point
- =m= to move the request at point
- =s= to send it
- =p= to preview it
- =e= to switch its environment
- =/= to filter the list
- =g= to refresh

Request and folder management stay file-based. Courier renames and moves normal
=.http= files and directories so Git continues to see plain filesystem changes.

** Response Mode

| Key       | Command                            | Description               |
|-----------+------------------------------------+---------------------------|
| =TAB=     | =courier-response-context-tab=     | Expand entry or move to next button |
| =RET=     | =courier-response-activate=        | Activate timeline item or open body |
| =C-c ?=   | =courier-dispatch=                 | Show response actions      |
| =h=       | =courier-response-show-headers=    | Show Headers tab          |
| =l=       | =courier-response-show-timeline=   | Show Timeline tab         |
| =o=       | =courier-response-open-body=       | Open body viewer          |
| =g=       | =courier-response-retry=           | Resend request            |
| =]=       | =courier-response-next-tab=        | Next response tab         |
| =[=       | =courier-response-prev-tab=        | Previous response tab     |
| =r=       | =courier-response-show-response=   | Show Response tab         |
| =t=       | =courier-response-show-tests=      | Show Tests tab            |
| =v=       | =courier-response-set-view=        | Select body view          |
| =V=       | =courier-response-toggle-pretty=   | Toggle raw/auto body view |
| =C-c C-k= | =courier-response-cancel=          | Cancel request            |

Body views currently include =auto=, =json=, =html=, =xml=, =javascript=,
=raw=, =hex=, =base64=, and =image=. Use =RET= or =o= to open the current body
in a dedicated viewer buffer; text-like views use an appropriate major mode,
and image views display the image when Emacs can render it. The main response
buffer also applies syntax highlighting to =json=, =html=, =xml=, and
=javascript= body views. The active top-level response view is shown in the
=header-line= as =Response >>=, =Headers >>=, =Timeline >>=, or =Tests >>=;
use =[= and =]= to cycle views, or the direct keys =r=, =h=, =l=, and =t=.
Use =C-c ?= in request, response, and overview buffers to open a transient menu
for the actions that make sense in the current Courier context.

* Response Buffer

The response buffer has four top-level views:

- *Response* shows the current response body. JSON is pretty-printed by default
  when possible, and =v= changes the body view while =V= toggles raw/auto.
- *Headers* shows response headers in a two-column layout.
- *Timeline* shows previous responses for the same request. It also has its own
  collapsed history list by default. Press =RET= on a history entry to expand
  it. The expanded entry is removed from the collapsed list below it. When
  expanded, Timeline behaves like a small section view:
  - *Request* shows the request URL, headers, and body snapshot that produced
    the selected history entry.
  - *Response* shows the selected historical response headers and body.
- *Network Logs* shows curl-level diagnostics such as argv, stderr, and exit
  code.
  Those section headings are buttons: use =RET= or mouse-1 on them to toggle
  sections open and closed. Use =M-x courier-response-clear-timeline= or the
  response transient menu to clear the current response buffer's timeline
  history, or press =RET= on the same history entry again to collapse the
  expanded view.
- *Tests* shows pass/fail indicators for all request =tests=.

The header line shows the current response view first, then status, duration,
and size; when visiting an older timeline entry it also shows the history
position (for example =2/5=).

* Customization

| Variable                      | Default   | Description                      |
|-------------------------------+-----------+----------------------------------|
| =courier-curl-executable=     | ="curl"=  | Path to curl                     |
| =courier-body-text-max-size=  | =524288=  | Max body size decoded as text    |
| =courier-default-collection-directory= | user home =courier/= | Default collection prompt root |
| =courier-default-request-method= | ="GET"= | Default method for new drafts    |
| =courier-default-timeout=     | =30=      | Default timeout in seconds       |
| =courier-history-max=         | =20=      | Max history entries per request  |
