#+title: Courier
#+author: Lucius Chen

Courier is a lightweight HTTP client for Emacs. It sends requests defined in
=.http= files using =curl= and displays responses in structured read-only
buffers.

* Features

- Edit HTTP requests as plain text in =.http= files
- Send requests with =C-c C-c= without blocking Emacs
- View responses in a single buffer with a header-line current-view indicator and an Emacs-native timeline inspector
- Variable interpolation with ={{name}}= syntax
- Built-in test assertions for status, headers, body, timing, and size
- Preview resolved requests before sending
- Select per-buffer environments from collection =env/*.env= files with =C-c C-e=
- Pretty-print JSON responses, with =v= to toggle raw view
- Start unsaved request drafts and place them into a collection on first save
- Open requests with =C-c C-o= from one annotated minibuffer picker, even
  outside an active Courier buffer
- Open environment files globally with =M-x courier-open-env=
- Import OpenAPI JSON or YAML into Courier collections with =M-x courier-import-openapi=
- Open a context-aware action menu with =C-c ?=

* Requirements

- Emacs 29.1 or later
- =curl= on PATH

* Installation

Clone this repository and add it to your =load-path=:

#+begin_src elisp
(add-to-list 'load-path "/path/to/courier")
(require 'courier)
#+end_src

* Quick Start

Create a file =example.http=:

#+begin_src http
+++
name = "Get User"

[body]
type = "json"

[auth]
type = "bearer"
token = "{{token}}"

[vars]
user_id = "42"
token = "abc123"

tests = ["status == 200"]
+++

GET https://httpbin.org/get/{{user_id}}?expand=profile
Accept: application/json
#+end_src

Open it in Emacs. Press =C-c C-c= to send.

* Collections

A Courier collection is a directory marked by =courier.json=. Courier stores
collections under a dedicated =collections/= directory inside
=courier-home-directory=. When =courier.json= is present, Courier treats that
directory as the collection root for request discovery and does not scan
environment files above it.

Example home layout:

#+begin_example
~/courier/
  collections/
    my-api/
      courier.json
      requests/
      env/
#+end_example

Minimal =courier.json= example:

#+begin_src json
{
  "name": "My API",
  "requestsDir": "requests",
  "envDir": "env",
  "defaultEnv": "local",
  "defaults": {
    "vars": {
      "base_url": "https://api.example.com"
    },
    "headers": {
      "accept": "application/json"
    },
    "auth": {
      "type": "bearer",
      "token": "{{token}}"
    },
    "timeout": 30,
    "follow_redirects": true
  }
}
#+end_src

Collection defaults are merged at request preparation time. Requests may also
declare nested =courier.json= files under =requests/= to provide folder-level
defaults. Merge order is:

- collection root defaults
- nested folder defaults
- selected environment file
- runtime vars written from prior responses
- request front matter =[vars]=
- request front matter =[vars.pre_request]=

Within that merge:

- later sources override earlier sources with the same key
- explicit request headers override inherited headers
- explicit request auth overrides inherited auth
- request settings such as =timeout= override inherited settings

* OpenAPI Import

Courier can import OpenAPI JSON or YAML into its own collection format:

- copied specs are stored under =courier-home-directory/specs/<collection>/=
- generated requests are stored under =courier-home-directory/collections/<collection>/requests/=
- the first server URL becomes collection default =base_url=
- path and method become Courier request files
- supported auth and body metadata are mapped into Courier front matter

Current scope:

- input: JSON and YAML
- auth mapping: =http bearer=, =http basic=, =apiKey=, and basic
  =oauth2 clientCredentials=
- body mapping: =application/json=, =application/xml=, =text/*=,
  =application/x-www-form-urlencoded=, =multipart/form-data=, and
  =application/octet-stream=

YAML import uses system Ruby with stdlib =psych= to normalize the spec into the
existing JSON import path.

Unsupported auth/body mappings are written to an import report under
=courier-home-directory/specs/<collection>/import-report.org=.

* File Format

A =.http= file contains one request using:

- optional TOML front matter between =+++= markers
- one raw HTTP request block after the front matter

Example:

#+begin_src http
+++
name = "Create User"
timeout = 10
follow_redirects = true

[body]
type = "json"

[auth]
type = "bearer"
token = "{{token}}"

[vars]
token = "abc123"
user_id = "42"

tests = ["status == 201"]

[scripts]
pre_request = """
(setq courier-script-request
      (plist-put courier-script-request :url "https://example.com/users/42"))
"""
+++

POST https://example.com/users?page=1
Accept: application/json
Content-Type: application/json

{
  "name": "Lucy"
}
#+end_src

Metadata belongs in TOML front matter:

| Key / Table           | Example                                | Description             |
|-----------------------+----------------------------------------+-------------------------|
| =name=                | =name = "Get User"=                    | Human-readable name     |
| =timeout=             | =timeout = 10=                         | Timeout in seconds      |
| =follow_redirects=    | =follow_redirects = true=              | Follow redirects        |
| =[body]=              | =type = "json"=                        | Body serialization type |
| =[auth]=              | =type = "bearer"=                      | Auth metadata           |
| =[vars]=              | =token = "abc123"=                     | Request-scoped vars     |
| =tests=               | =tests = ["status == 200"]=            | Test assertions         |
| =[scripts]=           | =pre_request = """..."""=              | Pre/post scripts        |

Method, URL, headers, and body stay in the raw HTTP block.

** Variables

Use ={{name}}= anywhere in the URL, headers, or body. Variables are defined
with the front matter =[vars]= table. Recursive expansion is supported up to 10
levels. Query parameters stay attached to the URL itself and use ={{var}}=
placeholders when needed.
If you prefer editing URL query params as key/value lines, use
=M-x courier-request-edit-params= or the request action menu. Courier opens a
small editor buffer and normalizes the result back into the request URL when
you apply it.

Courier resolves variable sources in this order:

- collection defaults from =courier.json=
- nested folder defaults from =requests/**/courier.json=
- selected environment file from =env/*.env=
- runtime vars previously extracted from responses
- request =[vars]=
- request =[vars.pre_request]=

That means request-local vars beat env values, and =pre_request= vars beat all
other variable sources.

** Body Types

=[body].type= controls how Courier interprets and sends the request body.

Supported types:

- =json=
- =xml=
- =text=
- =form-urlencoded=
- =multipart=
- =binary=
- =none=

Example:

#+begin_src http
+++
[body]
type = "form-urlencoded"
+++

POST https://example.com/session
Accept: application/json

username={{user}}&password={{password}}
#+end_src

Send-time behavior:

- =json= adds =Content-Type: application/json= when no explicit content type is present
- =xml= adds =Content-Type: application/xml= when missing
- =text= adds =Content-Type: text/plain; charset=utf-8= when missing
- =form-urlencoded= URL-encodes the body and adds
  =Content-Type: application/x-www-form-urlencoded= when missing
- =multipart= sends repeated =curl -F= parts derived from =[[body.parts]]=
- =binary= sends =--data-binary @path= and uses =[body].content_type= as the
  default content type when present and no explicit =Content-Type= header exists
- =none= requires an empty body and sends no request body

Use =M-x courier-request-set-body-type= or =C-c ?= in a request buffer to set
the body type. In the request section view, the active body type is shown in
the header line as =Body(JSON)=, =Body(XML)=, and so on.

** Auth

=[auth]= currently supports these runtime forms:

- bearer
- basic
- header
- api_key
- oauth2

Courier resolves auth values through the same variable expansion path as the
rest of the request. If an explicit header with the same name already exists in
the request, that explicit header wins.

Use =M-x courier-request-set-auth-type= or =C-c ?= in a request buffer to set
the auth type. The editor currently recognizes =none=, =bearer=, =basic=,
=header=, =api_key=, and =oauth2=. In the request section view, the active
auth type is shown in the header line as =Auth(None)=, =Auth(Bearer)=,
=Auth(Basic)=, =Auth(Header)=, =Auth(API Key)=, or =Auth(OAuth2)=.

=api_key= already works at send time in both =header= and =query= modes.
=oauth2= now works at send time for =grant_type = "client_credentials"=. Courier
first requests an access token from =token_url= using
=application/x-www-form-urlencoded=, then injects
=Authorization: Bearer <access_token>= into the main request. Missing tokens or
token endpoint failures surface as =OAuth2 Token Error= responses rather than
falling back silently.

** Scripts

Courier supports optional Elisp script blocks in front matter:

#+begin_src http
+++
[scripts]
pre_request = """
(setq courier-script-request
      (plist-put courier-script-request :url "https://example.com/from-script"))
courier-script-request
"""
post_response = """
(setq courier-script-response
      (plist-put courier-script-response :reason "Changed"))
courier-script-response
"""
+++

GET https://example.com
#+end_src

- =pre-request= runs after parsing and env loading but before validation and
  variable resolution.
- =post-response= runs after the response is parsed but before Courier runs
  =tests= assertions.

Scripts run as Elisp. The main bound variables are:

- =courier-script-request=
- =courier-script-response=
- =courier-script-env-vars=
- =courier-script-phase=

** Environment Files

Inside a collection, Courier reads environment files from the directory
configured by =courier.json= via =envDir=. The default is =env/= at the
collection root. Environment names come from filenames such as =local.env=,
=staging.env=, and =prod.env=. You may also use =.env= for an environment
named =default=.

If =defaultEnv= is set in =courier.json= and that environment exists, Courier
activates it automatically for the current request buffer. Otherwise, Courier
prompts with =completing-read= when multiple environments are available. The
choice remains buffer-local. Environment values are then merged with runtime
vars, request =[vars]=, and request =[vars.pre_request]= according to the
variable precedence described above. Opening an env file directly uses
=courier-env-mode=, which highlights comments and variable keys.

Example =env/local.env=:

#+begin_example
base_url=https://api.example.com
token=abc123
#+end_example

** Test Assertions

| Expression                          | Description                    |
|-------------------------------------+--------------------------------|
| =status == 200=                     | Status code equals             |
| =status != 404=                     | Status code not equals         |
| =header content-type contains json= | Header value contains string   |
| =body contains hello=               | Response body contains string  |
| =time < 500=                        | Response time under N ms       |
| =size < 102400=                     | Response size under N bytes    |

* Key Bindings

** Request Mode (=.http= files)

| Key       | Command                        | Description            |
|-----------+--------------------------------+------------------------|
| =C-c C-c= | =courier-request-send=         | Send request           |
| =C-c C-e= | =courier-request-switch-env=   | Switch environment     |
| =C-c C-j= | =courier-request-jump-section= | Jump to request section |
| =C-c C-m= | =courier-request-set-method=   | Change HTTP method     |
| =C-c C-n= | =courier-new-request=          | Create request draft   |
| =C-c C-p= | =courier-request-preview=      | Preview resolved form  |
| =C-c ?=   | =courier-dispatch=             | Show request actions   |
| =C-c C-l= | =courier-request-validate=     | Validate syntax        |
| =C-c C-o= | =courier-open=                 | Open request           |
| =C-c C-r= | =courier-rename-request=       | Rename current request |
| =M-.=     | =xref-find-definitions=        | Jump to variable definition |
| =C-x C-s= | =courier-request-save-buffer=  | Save draft or file     |

Use =M-x courier-new-request= or =C-c C-n= to open an unsaved draft buffer. The
draft buffer is temporary and the request editor itself starts at the request
line using =courier-default-request-method=, which defaults to =GET=. On the first
=C-x C-s=, Courier asks which collection should own the request. Courier keeps
collections under =courier-home-directory= in =collections/<name>/=. If you
type a collection name that does not exist yet, Courier creates it there
before saving the request into that collection's =requests/= directory. Courier
then asks for the request filename explicitly instead of inventing one from the
current request name; =.http= is added automatically unless you typed it
yourself.
The request-line method token is highlighted with a lightweight overlay so
=GET=/=POST=/etc. remain visually obvious while editing.
The request buffer uses a section view model with a navigation header line:

#+begin_example
Params  Body  Headers  >>
#+end_example

Use =C-c C-j= to jump to any section, including =Params=, =Body=, =Headers=,
=Auth=, =Vars=, =Script=, and =Tests=. The request line stays fixed as
the request identity block; the area below it is the active section view, not a
jump target into raw source text, and the current section is indicated in the
=header-line= rather than repeated as a second in-buffer title.

=Auth=, =Vars=, =Script=, and =Tests= are edited as TOML-backed fragments so
the visible section text matches the front matter Courier writes on save.

=Params= can still be edited directly as part of the URL, but the request-side
params views normalize those values into Courier params when you edit them.
Saving then writes them back into the request URL instead of keeping two
separate truths.

Use =M-x courier-create-collection= to create a collection explicitly. Use
=C-c C-o= to open another =.http= request from one annotated picker. Courier
scans known collections under =courier-home-directory=, lists request files
directly from the filesystem, and groups the picker by collection instead of
dropping into a directory prompt or parsing request contents just to build
picker entries.
Use =C-c C-e= to switch the active environment for the current request buffer.
Use =M-x courier-open-env= to open an environment file for editing from
anywhere; env candidates are grouped by collection, keep the env name as the
main label, and use the annotation for the env file path.
Use =M-.= in request buffers and Courier env buffers to jump to the effective
source-backed variable definition. Courier resolves definitions in request
buffers in this order: request =vars.pre_request=, request =vars=, selected
env, folder defaults, then collection defaults.
Use =C-c ?= in a request buffer for request actions. Request content itself
stays source-backed: Courier uses transient for actions, not as the primary
editor for params, auth, or script content.

Request and folder management stay file-based. Courier renames and moves normal
=.http= files and directories so Git continues to see plain filesystem changes.

** Response Mode

| Key       | Command                            | Description               |
|-----------+------------------------------------+---------------------------|
| =TAB=     | =courier-response-context-tab=     | Expand entry or move to next button |
| =RET=     | =courier-response-activate=        | Activate timeline item or open body |
| =C-c ?=   | =courier-dispatch=                 | Show response actions      |
| =o=       | =courier-response-open-body=       | Open body viewer          |
| =g=       | =courier-response-retry=           | Resend request            |
| =C-c C-j= | =courier-response-jump-tab=        | Jump to response view     |
| =v=       | =courier-response-set-view=        | Select body view          |
| =V=       | =courier-response-toggle-pretty=   | Toggle raw/auto body view |
| =C-c C-k= | =courier-response-cancel=          | Cancel request            |

Body views currently include =auto=, =json=, =html=, =xml=, =javascript=,
=raw=, =hex=, =base64=, =image=, and =document=. Use =RET= or =o= to open the
current body in a dedicated viewer buffer; text-like views use an appropriate
major mode, image views display the image when Emacs can render it, and PDF
documents open as file-backed viewers with =doc-view-mode= when available. The
main response buffer also applies syntax highlighting to =json=, =html=,
=xml=, and =javascript= body views. The =header-line= keeps =Response= as the
anchor view label; when =Headers=, =Timeline=, or =Tests= is active, that view
is inserted before =>>=, followed by the status summary. Use =C-c C-j= to jump
directly to a response view.
Use =C-c ?= in request and response buffers to open a transient menu for the
actions that make sense in the current Courier context.

* Response Buffer

The response buffer has four top-level views:

- *Response* shows the current response body. JSON is pretty-printed by default
  when possible, and =v= changes the body view while =V= toggles raw/auto.
- *Headers* shows response headers in a two-column layout.
- *Timeline* shows previous responses for the same request. It also has its own
  collapsed history list by default. Press =RET= on a history entry to expand
  it. The expanded entry is removed from the collapsed list below it. When
  expanded, Timeline behaves like a small section view:
  - *Request* shows the request URL, headers, and body snapshot that produced
    the selected history entry.
  - *Response* shows the selected historical response headers and body.
- *Network Logs* shows curl-level diagnostics such as argv, stderr, and exit
  code.
  Those section headings are buttons: use =RET= or mouse-1 on them to toggle
  sections open and closed. Use =M-x courier-response-clear-timeline= or the
  response transient menu to clear the current response buffer's timeline
  history, or press =RET= on the same history entry again to collapse the
  expanded view.
- *Tests* shows pass/fail indicators for all request =tests=.

The header line shows =Response= first, then the current secondary response
view when applicable, then status, duration, and size; when visiting an older
timeline entry it also shows the history position (for example =2/5=).

* Testing

Byte-compile the package:

#+begin_src sh
emacs --batch -L . -l courier.el -f batch-byte-compile courier.el
#+end_src

Run the full ERT suite:

#+begin_src sh
emacs --batch -L . -l courier.el -l ert -l test/courier-test.el --eval '(ert-run-tests-batch-and-exit t)'
#+end_src

The ERT suite includes local integration tests for:

- JSON echo requests
- multipart uploads
- binary uploads
- OAuth2 client credentials
- image and PDF responses

These integration tests start a local Python stdlib HTTP server from
=test/server/courier_test_server.py=. When =python3= is unavailable, they skip
instead of producing false positives.
